SmallOS network stack fixture

cserve on SmallOS

This larger static page is served by the vendored cserver ELF. It exists to give the kernel, VFS, TCP path, and userspace runtime a more meaningful response body than a tiny hello page.

Overview

/www/index.html

Server binary
ELF32

Built from third_party/cserver without modifying the upstream program.

Log destination
/logs

Background output should stay quiet while request logs go to cserve.log.

Purpose
smoke

Large enough to catch short reads, partial writes, and awkward buffering.

A static server fixture should do a little more than say hello. This page includes internal anchors, a table, form controls, preformatted text, nested sections, many repeated blocks, and enough CSS to make the asset non-trivial while still staying plain and portable.

The important compatibility marker remains present near the top of the document: cserve on SmallOS. Existing smoke tests can keep looking for that string while manual browser checks get something nicer to look at.

Behavior Matrix

These rows describe the pieces this page is useful for touching. They are written as normal content so the same bytes travel through the real static-file path instead of a test-only endpoint.

Area Expected observation Manual check Notes
ext2 lookup Server opens /www/index.html from the seeded volume. Run fsls www in the shell. Directory support is exercised before the server sends anything.
File read The full body reaches the HTTP client. Compare Content-Length to the local file size. A larger response helps reveal truncation and EOF mistakes.
TCP send Payload moves through several packets under QEMU user networking. Fetch with curl http://127.0.0.1:8080/. Useful with host forwarding such as port 8080.
Background process The shell prompt returns after bg apps/services/cserve --config cserve.ini. Make another request while typing in the shell. Request logs should not spill into the terminal.
Headers The response reports HTML content and a stable length. Run curl -i and inspect the first lines. This is also helpful when changing file metadata or stat behavior.
Parser tolerance Browsers handle the page with normal HTML5 behavior. Open the forwarded URL in a desktop browser. The document avoids external assets so every byte comes from one file.

Form Controls

This form is intentionally inert. It gives the browser a handful of common elements to render while keeping the server-side behavior as a simple static GET response.

What this is not

It is not a compatibility shim for cserver. It is just a better static asset for the OS image, so the operating system keeps owning the filesystem, networking, process, and syscall behavior.

That makes it a decent little canary for future kernel changes without coupling the page to the program implementation.

Representative Request

The page should be easy to inspect with a terminal client. This example is here as rendered text and as extra bytes in the response body.

  status: 200

Long Text Section

This section is intentionally verbose. Static serving bugs often show up only once a response crosses a boundary: one disk sector, one cluster, one TCP segment, one userspace buffer, or one loop in a send path. A realistic page gives those boundaries more chances to appear while still being simple enough to debug from serial output and curl.

A useful test fixture also needs recognizable structure. If the document ends halfway through a table row, a pre block, or a repeated card, the failure is visible before any tooling gets involved. If the document reaches the footer, the entire route from QEMU host forwarding to cserve, through the kernel socket layer, through the VFS, and back over TCP has done a decent amount of work.

This file is deliberately ordinary HTML. It does not rely on browser APIs, compression, chunked transfer encoding, external styles, or a build step. That keeps the signal clean: when something changes in the OS, this page can help show whether a plain static response still behaves.

Manual checks that work well with this page include opening the browser developer tools, checking the response length, reloading several times, requesting the direct index path, testing an unknown path for a 404, and watching that the shell prompt remains usable while cserve runs in the background.
